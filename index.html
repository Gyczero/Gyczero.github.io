产品实习总结

Posted on 2016-09-10   |  

第一个方案

第一个接触的方案是接机不选航班号。
简单的说，就是用户在选择接机确认行程的时候，填写航班号。此时后台找不到对应的航班，用户就无法选择接机服务，就会流失。这里要做的就是提供不填写航班号的接机服务方案。

一度

因为确实有用户这样反馈，刚开始接到这个需求的时候，我随即想那直接提供航班号、航站信息、时间和送达地不就行了吗。确定航班号后自动补充航站信息和时间，不填航班号也可以填航站信息和时间确认接机行程。
接着我开始想如果在单独的一个接机页面里，把这些信息都罗列出来的问题：
1.信息内容太多，一个接机页面就有航班号、航站信息、时间、送达地。接机是一个工具类的功能，目的应该是让用户最短的路径和时间完成自己的期望。这样信息全展开会增加时间和复杂性，体验不好。
2.信息有优先级，这里航班号的优先级应高于航站信息和时间，对用户来说只用输入几个字母即可，对司导来说有了航班号更容易安排自己的时间，可控性更高。
3.在选择接机服务的用户中搜不到自己的航班号的用户占比多少？这样设计到底会对多少用户有价值？

二度

我的数据驱动产品设计的思维从第三点开始萌发，向Mentor问了下用户占比，用户到达搜不到航班号页面的占比大约2%。有了这个数据，开始思考：
98%的用户是不需要不选航班号的设计的，同时鼓励用户输入航班号。所以最好在之前的功能逻辑上进行补偿而不是重构。

三度

进行补偿，所以我在搜不到航班号的页面中新加了选择，到达新的界面填写航站信息、时间和到达地。相当于新界面与原先提供航班号确认行程的界面同级，只是一种补偿方案。

逻辑闭环

有了这个方案之后去找Mentor，随即发现了自己的逻辑不完备，也感悟到了产品逻辑必须严密。
在我的方案中，因为确认行程的界面分为了两套，所以新加的那一套对原来的功能的影响，之后的操作页面信息结构的变化我都没有考虑。从确认行程 -> 确认订单 -> 提交订单 -> 订单详情 -> 订单操作整个的一个功能闭环都需要考虑，在之后的修改中，修改了部分页面的信息结构，保证新加方案的完整性，这块就不多说了。
基本完成方案后，我开始看业界是怎么做的。我喜欢先按照自己的想法构思功能方案，再看市场上竞品的方案。因为：
1.有创造感，也是做产品的初衷。
2.最初的想法最贴合当前的实际情况，每个公司的资源、环境包括上文中提到的数据等等各异，竞品的方案不一定适合自己。
这是去哪儿的接机方案：

一直想不通去哪儿的接机方案中航站楼为什么是可修改的。这里填写完航班号和日期会自动填充航站楼，修改航站楼也会清除航班号这个逻辑是完整的。但修改航站楼后航班号仍是必填选项，既然航班号是必填并且直接决定航站楼，航站楼信息应处在一个被动的层级，那为什么可以修改航站楼呢？

携程的接机方案：

携程的接机方案相比就比较清楚了，航班号在整个确认行程中起绝对支配作用。包括修改机场时必须先删除航班号，删除了主动信息，第二种根据机场时间接机的方案也是可行的。

对比了这些方案之后，想了下自己的设计。
1.携程去哪儿平台大，日活大，搜不到航班号的用户可能有较大的数量，所以它们干脆把所有的信息都放到首页，做成一套方案。
2.海外接送机对于携程去哪儿业务线来说只是一小块，内部开发、设计、测试资源有限，重构花费时间较多，所以考虑性价比，将这部分做成直接显示。

数据驱动

在实习的过程中一部分是做友盟埋点的梳理，虽然那几天做得有点崩溃= =,不过感觉埋点确实是熟悉一个产品设计流程的最好方式，清清楚楚地知道了一个产品模型的搭建。
埋点意义:
1.研究一个页面展示的事件数、来源以及一个页面各个按钮的点击数，改进产品设计
2.刻画用户画像，可以对用户下单时选择的车型进行统计，包括经济、舒适、豪华、奢华等
3.设置转化率漏斗，研究转化率下降严重的部分，改进下单支付流程和产品信息结构的设计
准备校招中，待续。。。

从“我的天气”看基于用户心智模型的设计

Posted on 2016-09-04   |  

早上起来看见外面刮风，想到已经立秋了，是不是该换件厚衣服。由于街上人少，不知道外面的人穿的什么，就想看今天的温度来决定换不换衣服。因为视觉方面较突出，我之前一直用“我的天气”这个APP，在看温度时，突然发现不同于iOS原生的天气应用，“我的天气“提供”昨天“的温度。被这么一个细节吸引，回忆自己之前看天气的情景，发现这样一个设计很精妙，通过用户的经历让用户更快速理解信息，更好地去影响用户的决策。

需求背后

分析早上的思考路径，不难发现，是从：
天气变化（诱因） -> 犹豫换厚衣服（决策犹豫） -> 街上人少（未寻找到参考系） -> 看天气 -> 对比昨天的温度(决策参考） -> 决定不换衣服
通常人们想要看温度的背后不仅仅是看温度，背后可能看是否换衣服，是否适合出行，晒被子等，这部分需求探索已经很完整了，在很多APP上都有体现。但在做决策时，我认为用户会分为两类：
1.对温度敏感用户
2.对温度不敏感用户
这里敏感是相对性的，在温度的小范围内的，比如15度和25度。字面上看都是在“较热”的范围，但用户的真实感知因人而异，在不经常看温度的用户心里，对温度小范围内高低敏感度偏弱。
